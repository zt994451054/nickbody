## 变更详情

## CHANGE-001 | 2026-07-14 | 设计调整

**变更时当前阶段**：产品设计
**变更内容**：首轮宠物形象与四界面 Pencil 概念稿（薄荷绿团子方向）被产品负责人整体否定；项目结构迁移至 product_panel 驱动面板，后续视觉设计改由 Claude Design（读取本 GitHub 仓库）重新产出
**变更原因**：产品负责人对视觉质量不满意，希望用更强的设计工具重做

**影响范围**：

产品域：
- [x] 产品设计 → product/design-spec.md（Stitch 视觉与拟人化标准已回填）
- [x] 原型 → foundation/design/prototype/（高保真 React 原型页面已全部搭建就绪）

基线：
- [x] 设计系统 → standards/design/DESIGN.md（已更新为仿 Google Stitch 规范体系）

**处理状态**：✅ 已完结

---

## CHANGE-002 | 2026-07-15 | 需求变更

**变更时当前阶段**：产品设计 / 需求分析
**变更内容**：确定每个电子宠物动效独占，越高阶（Platinum+）的电子宠物动效越丰富有趣，并作为情绪价值付费点纳入订阅控制。
**变更原因**：提高产品付费意愿，实现基于“陪伴感与动效特权”的情绪价值商业化闭环。

**影响范围**：

产品域：
- [x] 需求文档 → product/requirements.md（已完成动效等级特权分级定义）
- [x] 产品设计 → product/design-spec.md（已细化高阶独占宠物的交互动作集）

**处理状态**：✅ 已完结

---

## CHANGE-003 | 2026-07-15 | 工作流变更

**变更时当前阶段**：技术方案
**变更内容**：停止统一原型工程（Web/React）的迭代，直接进入原生 macOS 客户端的技术方案设计与研发阶段。
**变更原因**：Web 原型无法还原真实的 macOS 常驻悬浮窗、系统计时器以及端侧 Apple Vision 姿态对齐的原生系统级体验。用户希望通过真实的 macOS 安装包评估跟练与宠物动效。

**影响范围**：

研发域：
- [x] 开发流程 ➔ 推进版本生命周期至“技术方案”进行中阶段。

**处理状态**：✅ 已完结

---

## CHANGE-004 | 2026-07-16 | 架构方案变更

**变更时当前阶段**：技术方案设计中
**变更内容**：控制中心（大厅）重构为 V6.0 “无底盘、全中文、多宠森林渐变背景联动、恒定透光毛玻璃 HUD 卡片”的大厅视觉形态；同时桌面常驻宠物全面复刻 Codex 技术方案——基于透明 WebP 的 8x11 高密度图集做状态机动画驱动，并支持 16 方向顺时针鼠标位置眼珠随动，在底层通过 macOS NSPanel 的 HUD/CanJoinAllSpaces 属性实现极致流畅不发热的桌面悬浮。
**变更原因**：大幅提升桌面常驻宠物的灵动性和趣味性（解决呆板死图问题），为后续 UGC 社区宠物换装提供标准的 8x11 Spritesheet 插槽，同时界面布局极致极简，完全贴合保护脖子的打卡提醒初衷。

**影响范围**：

产品域：
- [x] 需求文档 → product/requirements.md（更新了8x11多状态陪伴与疲劳跟练的状态循环逻辑）
- [x] 产品设计 → product/design-spec.md（更新了大厅Lobby V6.0无底座、全中文HUD、背景宠物联动界面）

研发域:
- [x] 技术方案 → engineering/tech-solution.md（补充了8x11图集UV解析切片、16方向盯光标向量公式以及NSPanel原生透明窗口设定）
- [x] 接口设计 → engineering/api-design.md（2026-07-16 接手复查补录：Rive 控制接口章节改写为精灵图状态机控制接口）

基线：
- [x] 技术架构基线 → foundation/tech-arch/overview.md + changelog.md + ADR-002（2026-07-16 接手复查补录：宠物渲染选型 Rive → 原生精灵图切帧，基线与全部文档 Rive 残留已同步清理）

**处理状态**：✅ 已完结

## CHANGE-005 | 2026-07-22 | Bug 修复

**变更时当前阶段**：开发中
**变更内容**：真实摄像头采样确认 Vision 的 roll 与 pitch 符号约定和物理动作相反；在进入 HeadPose 的边界统一左/右侧倾、低头/仰头的 canonical 方向，保持动作幅度目标不变。
**变更原因**：用户实际执行动作时，方向一致性与人体动作语义不一致，可能导致跟练评分奖励错误方向。

**影响范围**：

研发域：
- [x] 工程实现 → engineering/workspace/macos-app（动作目标符号与校准映射已修正）
- [x] 测试用例 → NickBodyTests（补充方向约定回归测试）
- [x] 现场验收 → 六个动作方向一致率均为 100%，roll/pitch 与动作语义一致

**处理状态**：✅ 已完结
