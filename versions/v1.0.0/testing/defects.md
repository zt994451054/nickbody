## 缺陷状态流转

```
⏳ 待修复 → 🔄 修复中 → ✅ 已修复 → 🕐 待验收 → ✅ 验收通过（已关闭）
                                              ↘ ❌ 验收失败 → 返回「待修复」
🚫 豁免（仅 P1，经产品负责人确认；P0 不可豁免）
```

| 状态 | 说明 |
|------|------|
| ⏳ 待修复 | 缺陷已确认，等待研发修复 |
| 🔄 修复中 | AI 正在修复 |
| ✅ 已修复 | AI 已完成修复，触发人工验收 |
| 🕐 待验收 | 等待人工验收 |
| ✅ 验收通过 | 人工验收合格，缺陷关闭 |
| ❌ 验收失败 | 人工验收不合格，退回待修复 |
| 🚫 豁免 | 仅 P1 可豁免，须产品负责人确认；**P0 不可豁免** |

## 说明

- **新增缺陷**：同步更新 ①「缺陷列表」表新增一行；②「缺陷统计」表；③关联用例状态改为 ❌ 失败
- **修复更新**（AI）：在「修复历程」节追加记录，更新修复次数和最近修复时间，状态改为 ✅ 已修复 → 🕐 待验收
- **验收更新**（人工）：更新「人工验收」节，同步更新「缺陷列表」验收状态和「缺陷统计」表
- **工件上传**：截图/录屏/日志按 SOP-10 上传 OSS（路径 `versions/{ver}/testing/`，命名 `YYYYMMDD-BUG{序号}-描述.ext`，见 OSS.md），CDN URL 填入「工件 URL」字段
- **hotfix 场景**：发现版本与修复版本不同（如 v1.0.0 发现 → v1.0.1 修复），「修复版本」填实际 hotfix 版本号
- **P0 不可豁免**：必须本版本关闭，不可延迟修复
- **修复影响接口/DB/需求**：推进到「已发布」时按 §4.2 合入基线（SOP-07 / SOP-11）
- 变更追踪 → [CHANGES.md](../CHANGES.md)
