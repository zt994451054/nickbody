# v1.0.0 验收计划与用例

> 验收是版本正式交付前在**生产环境**进行的最终质量确认。
> 区别于测试（staging，目的是"发现问题"），验收的目的是"确认可以交付给用户"。
>
> **触发时机**：发布部署完成（release.md 全部步骤执行完毕）后执行。
> **执行者**：AI 按用例执行，人工做最终验收判断。
>
> 验收用例来源：综合 `test-cases.md`（核心链路）+ `defects.md`（已修复的关键 bug）+ `CHANGES.md`（本版本重要变更），由 AI 精简生成，需产品负责人确认范围。

## 验收范围

> 验收用例应聚焦核心链路，标准高于测试用例（生产环境，面向真实用户）。
> 产品负责人须确认以下来源中哪些纳入验收，哪些不必验证。

| 来源 | 说明 | 是否纳入 |
|------|------|---------|
| 核心用户旅程 | 用户最关键的端到端流程（来自 test-cases.md P0 用例） | 全部纳入 |
| 本版本新功能 | 本版本新增的主要功能（来自 test-cases.md 标注「已确认」的需求） | 全部纳入 |
| 已修复关键 bug | P0/P1 缺陷修复的回归验证（来自 defects.md） | 全部纳入 |
| 本版本重大变更 | CHANGES.md 中影响范围较大的变更 | 视情况纳入 |
| 非核心功能 | P2/P3 功能或边界场景 | **不纳入**，遗留已知 |

## 验收退出条件

| 条件 | 标准 |
|------|------|
| P0 验收用例 | 100% 通过 |
| P1 验收用例 | 100% 通过，或有产品负责人明确的遗留决策 |
| 发现新 P0 bug | 必须修复后重新验收 |
| 发现新 P1 bug | 视严重程度，由产品负责人决定是否 hotfix 再验收 |
| 版本最终交付 | 产品负责人在「验收报告」节签收确认 |

## 说明

- 验收用例由 AI 综合生成，**产品负责人须确认验收范围**后方可开始执行
- AI 执行验收用例时，须使用生产环境凭证（见上方「验收凭证」节）
- 执行失败时，按严重程度决定处理方式：
  - P0 新 bug → 立即记录到 `defects.md`，触发 SOP-11 hotfix，修复后重新验收
  - P1 新 bug → 由产品负责人决定是否 hotfix，决策记录在「遗留问题说明」
  - P2/P3 新 bug → 记录到 `defects.md`，下版本修复，遗留说明中注明
- 工件上传路径：`versions/{ver}/testing/`，命名 `YYYYMMDD-AC{序号}-描述.ext`（见 OSS.md）
- **版本交付完成 = 产品负责人在「版本交付确认」行签收**，AI 随后执行 SOP-07 将版本阶段标记为「已发布」
- 变更追踪 → [CHANGES.md](../CHANGES.md)
