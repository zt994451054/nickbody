### 与本地开发环境的隔离策略联动

测试执行产生的数据须遵循 `engineering/docs/local-env.md §4` 的 per-developer 隔离策略：

- MySQL 测试数据写入自己的 `app_dev_${USER}` schema（或专用测试 schema）
- Redis key 带 `${USER}:` 前缀，避免与他人测试数据混淆
- MQ topic 带 `${USER}.` 前缀

禁止在无前缀的 dev 共享路径写测试数据，避免污染他人开发环境。

## 进入条件

> 以下条件全部满足后，方可开始正式测试

- [ ] 本版本所有测试范围内的功能需求状态已更新为「已确认」（见 requirements.md）
- [ ] 本版本所有功能已开发完成并部署至 staging 环境
- [ ] 数据库变更（db-design.md 增量 DDL）已在 staging 环境执行
- [ ] 配置变更（tech-solution.md 配置变更需求）已在 staging 环境生效
- [ ] 研发单元测试通过（CI 流水线绿色）
- [ ] API 设计文档（api-design.md）已与实现核对一致

## 退出条件（测试通过标准）

| 条件 | 标准 |
|------|------|
| 测试用例执行率 | P0 用例 100% 执行，P1 用例 ≥ 90% 执行 |
| 用例通过率 | P0 用例 100% 通过，P1 用例 ≥ 95% 通过 |
| P0 缺陷 | 全部关闭 |
| P1 缺陷 | 全部关闭，或经产品负责人明确确认豁免（豁免决策须记录在对应缺陷文件的「豁免说明」字段中） |
| P2 缺陷 | 可遗留，在测试报告中记录 |
| 回归测试 | 与本版本有交叉的核心流程无新增缺陷 |
| 性能测试 | 如 requirements.md 有性能要求，接口响应时间达标率 ≥ 99%；无性能要求则此条不适用 |
| 测试报告 | 已完成并结论明确 |

## 风险与应对

| 风险 | 可能影响 | 应对方案 |
|------|---------|---------|
| 外部依赖（如短信网关）在 staging 不可用 | 验证码相关用例无法正常执行 | 使用 mock 短信网关，在代码层 stub 短信发送 |
| staging 环境不稳定 | 测试结果不可信 | 发现环境问题及时上报，等待修复后重新执行 |

- 本测试计划基于 requirements.md 中「已确认」状态的需求，草稿需求不纳入测试范围
