# 需求池（Backlog）

> 本文件是跨版本的需求收集与管理池，记录所有已提出的需求点及其状态。
> 与版本内 `versions/{ver}/product/requirements.md` 的关系：
> - **Backlog**：跨版本，粗粒度，一行一个需求点，记录"有哪些需求"
> - **版本需求文档**：版本内，细粒度，展开业务规则/验收标准，记录"这个版本怎么做"

---

## 需求状态说明

| 状态 | 含义 |
|------|------|
| 待评审 | 需求已提出，尚未进行评审 |
| 已确认 | 评审通过，需求合理，等待纳入版本 |
| 已纳入版本 | 已分配到具体版本，正在或已完成开发 |
| 已拒绝 | 评审不通过，不纳入产品 |
| 已推迟 | 暂缓，未来版本重新评估 |

> ⚠️ **「已确认」≠「已纳入版本」**：已确认只说明需求合理，未分配版本。已确认未分配的需求会在版本规划时统一评估。

## 说明

- **提需求**：人提出需求，AI 整理到本文件的「需求池」表，编号自动递增（SOP-00.5）
- **评审**：人更新评审状态（待评审 → 已确认 / 已拒绝 / 已推迟）
- **版本规划**：启动新版本时，从「已确认」需求中选取纳入，AI 更新「纳入版本」字段并将状态改为「已纳入版本」（SOP-01）
- **需求展开**：纳入版本后，在 `versions/{ver}/product/requirements.md` 中详细描述业务规则和验收标准
- 已拒绝/已推迟的需求移入「归档」表，不删除，保留决策记录
- 变更追踪 → [../../versions/{ver}/CHANGES.md]
